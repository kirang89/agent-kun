# Test Harness Reference Table

*Derived from SQLite's testing methodology — generalized for any software system.*

| # | Category / Type | Intent (What It Tests For) | Mechanism & Tooling | Example Scenarios | Complexity |
|---|---|---|---|---|---|
| **1** | **Unit / Regression Tests** | Correctness of individual modules and functions through the public API; ensures past bugs never recur. | Write scripted test cases that call public APIs with known inputs and assert expected outputs. Every fixed bug gets a dedicated regression test before close. **Tools:** language-native test frameworks (pytest, JUnit, Go `testing`, TAP/TCL), CI runners. | • Parse a config file and verify all fields are populated correctly. • Serialize an object to JSON and back, assert round-trip equality. • Call `createUser()` with duplicate email, expect a conflict error. • Reproduce issue #472: passing a negative offset no longer returns stale data. | Low |
| **2** | **Parameterised / Matrix Testing** | Correct behaviour across all supported build configurations, compiler flags, platforms, and runtime options. | Define a configuration matrix (OS, arch, compiler, feature flags, 32/64-bit, endianness) and run the full suite under each combination. Automate with scripts that generate and drive each variant. **Tools:** CI matrix strategies (GitHub Actions `matrix`, Jenkins pipelines), cross-compilation toolchains, Docker multi-arch. | • Run the test suite on Linux arm64, macOS x86_64, and Windows x64. • Build with TLS enabled and disabled, verify both pass. • Test with GCC 13 and Clang 18 at `-O0` and `-O2`. • Run under musl libc and glibc to catch POSIX assumption differences. | Medium |
| **3** | **Fault Injection — Out-of-Memory** | Graceful degradation and clean error reporting when memory allocation fails at every possible call-site. | Interpose a custom allocator that fails on the *N*-th allocation. Run the operation in a loop, incrementing *N* each iteration until it succeeds without fault. After each injected failure, verify the system reports a clean error and leaks nothing. **Tools:** `failmalloc`, custom `LD_PRELOAD` shims, language-level allocator hooks, Valgrind `--malloc-fill`. | • Fail malloc during image resize — verify partial output is cleaned up. • Fail allocation while building an in-memory index — verify the error propagates and no half-built index remains. • OOM during connection pool expansion — verify existing connections are unaffected. | High |
| **4** | **Fault Injection — I/O Errors** | Correct handling of disk/network I/O failures at every possible I/O call-site. | Interpose a virtual filesystem or I/O layer that returns errors on the *N*-th operation. Loop through all failure points; after each, verify the system surfaces a meaningful error and remains in a consistent state. **Tools:** `libfiu` (fault injection in userspace), `strace`-based replay, FUSE-based fault filesystems, custom VFS layers. | • Fail a disk write mid-transaction — verify the write is rolled back cleanly. • Return `EIO` on the 3rd `read()` while streaming a file — verify the client gets an error, not a partial response. • Simulate network timeout during an API call — verify retry logic activates and no duplicate side-effects occur. | High |
| **5** | **Crash / Power-Loss Recovery** | Data integrity and atomicity guarantees surviving unclean shutdowns at arbitrary points during writes. | Simulate a crash by snapshotting filesystem state after selected write operations, then restoring the snapshot with realistic post-crash damage (reordered writes, partial pages, zeroed sectors). Restart the system and verify it recovers to a consistent state, then run an integrity check. **Tools:** `dm-fuse` / `CharybdFS` / custom VFS, `kill -9` with journaled FS snapshots, container checkpointing (CRIU). | • Kill the process mid-write to a WAL file — restart and verify committed data is intact. • Simulate power loss during a two-file atomic swap — verify either both or neither file is updated. • Crash during compaction of a log-structured store — verify restart recovers without data loss. | High |
| **6** | **Differential / Cross-Implementation Testing** | Semantic correctness by comparing outputs against independent reference implementations. | Run the same large corpus of inputs through your system and one or more reference implementations. Compare outputs deterministically (sort results, hash large outputs). Flag any divergence as a potential bug. **Tools:** `sqllogictest` (for query engines), custom diff harnesses, property-based test generators. | • Run 10k math expressions through your evaluator and Python `eval()`, diff results. • Feed the same HTTP requests to your server and `nginx`, compare response headers and status codes. • Compare your date-parsing library against `dateutil` on 50k date strings across formats and time zones. | Medium |
| **7** | **Fuzz Testing** | Robustness against malformed, unexpected, or adversarial inputs — surfaces crashes, hangs, assertion failures, and security vulnerabilities. | Use coverage-guided mutational fuzzing on both the primary input format and any persisted file/data format simultaneously (structure-aware fuzzing). Maintain a corpus of historically interesting inputs and replay them in CI as regression fuzz cases. **Tools:** `libFuzzer`, `AFL++`, `OSS-Fuzz`, `Honggfuzz`, `cargo-fuzz`; structure-aware with custom mutators or grammar-based fuzzers (`Grammarinator`). | • Fuzz the JSON parser with random byte streams — no crash or hang allowed. • Mutate valid protobuf messages and feed to the deserialiser — verify graceful rejection. • Fuzz the image decoder with mutated PNGs — verify no buffer overflows (under ASan). • Simultaneously mutate both the config file format and the API input to test interactions. | Medium–High |
| **8** | **Boundary Value & Edge-Case Testing** | Correct behaviour at value extremes — max/min integers, empty inputs, maximum sizes, off-by-one conditions. | Instrument source with boundary-coverage macros at every conditional that tests a limit. Write test cases that hit both sides of each boundary. Verify coverage tooling confirms each boundary branch was exercised. **Tools:** `gcov -b`, `llvm-cov`, custom `testcase()` / `assert()` macros in source. | • Insert exactly `MAX_ITEMS` and `MAX_ITEMS + 1` into a collection — verify limit enforcement. • Send a request with a 0-byte body, a 1-byte body, and a body at the max size limit. • Pass `INT64_MAX`, `INT64_MIN`, `0`, and `-1` to every numeric API parameter. • Test an empty string, a single-character string, and a string at the max-length boundary. | Medium |
| **9** | **Code Coverage Verification** | Assurance that the test suite exercises every branch (and ideally every boolean sub-condition — MC/DC) of the shipped code. | Compile with coverage instrumentation, run the full suite, then measure. Target 100% branch coverage for core code. Importantly, re-verify coverage under *release build flags* (not just debug) to ensure the as-shipped binary has identical behaviour. **Tools:** `gcov -b`, `lcov`, `llvm-cov`, `coverage.py`, `JaCoCo`, `istanbul/c8`. | • Generate an HTML coverage report and flag any core module below 95% branch coverage. • Verify that a newly added error-handling branch is exercised by at least one test. • Re-run coverage under `-O2` release flags and diff against the debug coverage report. • Check that every `if/else` in the authentication module has both branches hit. | Medium |
| **10** | **Mutation Testing** | Validates that the test suite actually detects real defects — not just that it covers code, but that it *fails* when code is broken. | Systematically inject small semantic mutations into compiled code or source (flip branches, change operators, remove statements). Verify the test suite catches each mutation. Surviving mutants indicate weak test coverage. **Tools:** `mutmut` (Python), `pitest` (Java), `mull` (C/C++ LLVM-based), `Stryker` (JS/.NET), manual binary patching. | • Change `>=` to `>` in a bounds check — verify a test fails. • Replace `&&` with `||` in a validation guard — verify a test catches it. • Remove a `return` statement from an early-exit path — verify a test detects the behavioural change. • Negate a boolean flag in the caching layer — verify a test surfaces incorrect cache hits. | High |
| **11** | **Static Analysis** | Detect latent bugs, undefined behaviour, type errors, and style violations without executing code. | Run multiple static analysers as part of CI. Compile with maximum warnings enabled and treat warnings as errors. Use at least two independent tools to catch what each misses. **Tools:** `-Wall -Wextra -Werror`, Clang `scan-build`, `cppcheck`, `clang-tidy`, `Coverity`, `SonarQube`, language linters (`eslint`, `clippy`, `mypy`). | • Flag an uninitialised variable read in a rarely-taken error path. • Detect a potential null-pointer dereference after a failed lookup. • Catch an unreachable `default` case in a switch missing a new enum variant. • Identify a format-string mismatch in a logging call. | Low |
| **12** | **Dynamic Analysis — Memory & UB** | Detect memory leaks, use-after-free, buffer overflows, data races, and undefined behaviour at runtime. | Run the full test suite under memory and UB sanitizers. Use multiple compilers' sanitizers and also standalone tools. Test with both signed and unsigned char defaults, and with different optimization levels. **Tools:** `Valgrind` (memcheck, helgrind, DRD), `-fsanitize=address,undefined,thread,memory` (ASan/UBSan/TSan/MSan), MSVC `/RTC1`, `-ftrapv`. | • Run the test suite under ASan — detect a heap-buffer-overflow in the parser. • Run under UBSan — catch a signed integer overflow in a hash function. • Run under MSan — find a read of uninitialised memory in a deserialiser. • Run under Valgrind — detect a 64-byte leak on a rarely-taken error path. | Low–Medium |
| **13** | **Integrity / Consistency Self-Checks** | Verify that internal data structures remain self-consistent after every operation, especially after error paths. | Build self-check commands into the system (analogous to `fsck`). Run these checks after every fault-injection and crash-recovery test. Expose them as user-facing diagnostic commands too. **Tools:** Custom validation routines, `assert()` instrumentation (thousands of in-code assertions), checksums on internal structures. | • After every B-tree insertion, verify parent-child pointer consistency. • After crash recovery, run a full checksum over all stored records. • Validate that an in-memory cache and its on-disk backing store agree after every write. • Assert that a reference count never goes negative after any sequence of acquire/release calls. | Medium |
| **14** | **Disabled-Optimisation Testing** | Prove that optimisations don't change observable behaviour — the fast path and the slow path must produce identical results. | Provide runtime switches to disable each optimisation individually. Run the full input corpus with optimisations on and off, then diff the results. Any divergence is a correctness bug. **Tools:** Runtime feature flags, A/B diff harnesses, custom test-control interfaces. | • Disable the query-result cache and verify all API responses are byte-identical. • Turn off SIMD-accelerated encoding — verify output matches the scalar path. • Disable connection pooling — verify request results are unchanged. • Turn off read-ahead prefetching — verify all reads return the same data. | Medium |
| **15** | **Malformed / Corrupt Input Testing** | Resilience against damaged, truncated, or adversarially crafted persistent data (files, databases, caches). | Feed the system hundreds of intentionally corrupted variants of its persisted format — bit-flips, truncations, header overwrites, cross-version files. Verify the system either rejects them with a clear error or recovers gracefully, never crashing or leaking data. **Tools:** `zzuf`, `radamsa`, custom binary mutators, hand-crafted corpus of known-evil files. | • Truncate a data file to half its size — verify the system reports corruption, doesn't crash. • Flip random bits in the file header — verify graceful rejection. • Feed a v1-format file to a v3 reader — verify it detects the version mismatch. • Zero out the checksum block of a snapshot file — verify integrity check catches it. | Medium |
| **16** | **Resource Leak Detection** | Ensure no file handles, sockets, memory, or OS resources leak under any code path, including error paths. | Run the full suite with leak-detecting allocators and file-descriptor tracking. Verify that every allocation is paired with a deallocation, even when errors are injected. **Tools:** `Valgrind --leak-check=full`, ASan leak detection, `lsof` / `/proc/pid/fd` monitoring, OS-level resource accounting. | • Open and close 10k connections in a loop — verify fd count returns to baseline. • Trigger an error mid-transaction — verify temp files are cleaned up. • Run a long-lived workload — monitor RSS over time for steady-state growth (indicates a slow leak). • Fail authentication on every attempt for 1k iterations — verify no socket handles leak. | Low–Medium |
| **17** | **Concurrency / Multi-Process Stress Testing** | Correctness under concurrent access — race conditions, deadlocks, lock contention, data corruption from parallel writers. | Launch multiple threads/processes performing randomised read/write workloads simultaneously. Vary timing with sleeps and scheduling hints. Run under thread sanitizers and verify data integrity after the stress run. **Tools:** TSan (`-fsanitize=thread`), `stress-ng`, custom multi-threaded test drivers, `helgrind`. | • 50 threads writing to the same key-value store — verify no lost updates. • Readers and writers operating concurrently — verify readers never see a partially-written record. • Two processes competing to acquire a file lock — verify mutual exclusion holds. • Randomised mix of create/delete/list operations under 100 goroutines — verify final state is consistent. | High |
| **18** | **Defensive / Unreachable Code Verification** | Confirm that code paths believed to be unreachable truly are, and that defensive guards don't mask real bugs. | Wrap defensive branches in macros (e.g., `ALWAYS(x)` / `NEVER(x)`) that, in testing, assert the expected condition holds. In release builds, the macro is a harmless pass-through. If an assertion fires in testing, the "impossible" case is actually reachable — it's a real bug. **Tools:** Custom macros, `assert()`, `__builtin_unreachable()`, compiler warnings for dead code. | • A `default` case in a switch that "can't happen" — the macro asserts it truly never fires across the full suite. • A null-check after a call that "always returns non-null" — verify the assertion never triggers. • A fallback branch guarding against negative array indices — confirm no test ever reaches it. | Low |
| **19** | **Regression Test Discipline** | Guarantee that every historically reported bug is permanently guarded against re-introduction. | Enforce a process rule: no bug may be marked as fixed without a committed test case that reproduces it. Maintain a tagged corpus of these tests so they are never accidentally pruned. **Tools:** Issue-tracker integration, CI gating on test count, code-review checklists. | • Bug #301: unicode normalisation edge case — test pinned with issue tag, runs on every CI build. • Bug #587: race condition on shutdown — dedicated stress test added, linked to the fix commit. • CI gate rejects any PR that decreases the total test count. • Quarterly audit of closed bugs vs. test coverage to catch gaps. | Low |

## Key Meta-Principles

| Principle | Description |
|---|---|
| **Independent harnesses** | Use multiple independently-developed test frameworks. Bugs in one harness are unlikely to exist in another. |
| **Test the shipped artefact** | Re-run coverage and correctness tests against the *release* build (with production compiler flags), not just debug builds. |
| **Assume the test suite is wrong too** | Use mutation testing and coverage analysis to test *the tests themselves*. |
| **Anomaly follow-up** | After every fault-injection or crash test, run a full integrity/consistency self-check — not just "did it not crash." |
| **Corpus permanence** | Interesting fuzz inputs, regression cases, and differential test corpora are saved permanently and replayed in CI. |
